So if a read and write occurs at the same time to the same page, and no safety measures are enforced, it's possible for the data that is still being written to be read.
A simple way to fix this is to enforce a lock, or mutex, so that the first transaction locks down the page, and the second one must wait until the first one finishes and releases the lock.
However, that could be slow, especially if you have a very high read volume that constantly locks down a bunch of pages and blocks writes.
Therefore, both DB and LMDB use a copy-on-write, or more specifically shadow-paging approach, to circumvent this.
If a page is being read and we want to write to it, rather than writing to the in-use page, we instead make a shadow copy and write to the copy, which is marked as the current database state.
Then, when the copy is being read, writes will reclaim the memory space of the original and apply the changes there.
If a read comes in during a write, it will simply read the page that is not currently in use.
So, in this way, reads never block writes, and writes never block reads.
